# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

Immich AutoTag is a rule engine that applies batch changes to [Immich](https://immich.app) photo/video libraries: creates and assigns albums, generates and assigns tags, fixes asset dates, and manages album permissions — all driven by a user-defined configuration file.

## Essential commands

```bash
# Set up environment (creates .venv, installs deps, generates immich-client from OpenAPI)
source setup_venv.sh
# Add --clean to force client regeneration; --dev to install dev/quality-gate tools
source setup_venv.sh --clean --dev

# Run the application
bash run_app.sh
# or
python -m immich_autotag

# Run Quality Gate (linting, type checks, architecture checks)
bash scripts/devtools/quality_gate_py/venv_launcher.sh --level=STANDARD --mode=CHECK

# Release a new patch version (bumps pyproject.toml, tags, pushes — triggers GitHub Actions)
bash scripts/devtools/release.sh 0.80.X
```

## CI/CD

- **Jenkins** (local): runs on every push to active branches. Pipeline stages: clean caches → setup venv → validate imports → Quality Gate → run application. Config: `Jenkinsfile`. Logs archived under `logs_local/`.
- **GitHub Actions** (`.github/workflows/release.yml`): triggered by version tags `v*.*.*`. Publishes to PyPI and builds/pushes Docker images (`txemi/immich-autotag:latest` and `txemi/immich-autotag:cron`).
- **Release workflow**: tag from the feature branch → GitHub Actions handles PyPI + Docker → merge to `main` once CI is green.

## Architecture

### Processing pipeline (`immich_autotag/entrypoints/main_logic.py`)

```
init_config_and_logging()
  → assert_client_server_version_match()
  → init_collections_and_context()        # loads all albums/assets from Immich API
  → delete_unhealthy_temp_albums()        # maintenance
  → apply_conversions_to_all_assets()     # batch tag/album renames (optional)
  → process_permissions()                 # album sharing rules
  → process_assets_or_filtered()          # classification, date correction, duplicate sync
  → finalize()                            # write logs, stats, links markdown
```

### Module dependency rules

- `immich_autotag/albums/` — core album logic. **Must not** depend on `assets/`.
- `immich_autotag/assets/albums/` — asset↔album business logic. May use `albums/` but not the reverse.
- `immich_autotag/assets/albums/temporary_manager/` — isolated temporary album lifecycle.
- `immich_autotag/common/` — generic utilities with **no** imports from any other `immich_autotag` subpackage. Treat as an independent library.
- `immich_autotag/config/` — configuration loading. `models.py` defines all Pydantic models; `internal_config.py` holds derived/internal flags.
- `immich_autotag/api/` — thin wrapper around the auto-generated `immich-client/` (generated from Immich's OpenAPI spec via `setup_venv.sh`).

### Tag naming convention

Tags use underscores `_` only — no slashes `/`. Hierarchical tags are unreliable in the Immich API.

- **Input tags** (`autotag_input_*`): assigned manually by the user to signal intent to the rule engine.
- **Output tags** (`autotag_output_*`): written by autotag to report classification results. Never assign these by hand.

## Configuration

Users configure the tool via a file at `~/.config/immich_autotag/config.yaml` (YAML) or `~/.config/immich_autotag/config.py` (Python). Two formats are supported:

- **YAML** (`immich_autotag/config/user_config_template.yaml`): recommended for end users.
- **Python** (`immich_autotag/config/user_config_template.py`): richer inline docs, used for development. The Python template is the source of truth for documentation — `description=` fields on Pydantic objects are intentional, as they propagate into runtime output (e.g. CLAUDE.md generation from `model_dump()`).

All Pydantic config models live in `immich_autotag/config/models.py`. Every model has an optional `description` field that serialises with `model_dump()`.

**Never commit a config file with real credentials.** The templates contain only placeholder values.

## Versioning

- Source of truth: `pyproject.toml` → version must always match the latest git tag.
- `immich_autotag/version.py` is auto-generated by `scripts/devtools/update_version_info.sh`; do not edit by hand.
- `scripts/devtools/release.sh <version>` automates the full bump+tag+push flow.
- PATCH increments are used for Quality Gate milestones with no functional changes; MINOR for new features; MAJOR for breaking changes.

## Branch strategy

- `main` — stable, production-ready.
- `develop` — integration branch.
- `docs-track` — documentation-only changes (no code); permanent, do not delete.
- Feature branches use prefixes: `feature/`, `fix/`, `refactor/`, `perf/`, `improvement/`.

## Issue tracking (`docs/issues/`)

Working documentation for features, bugs, and design decisions lives under `docs/issues/`.

- **Registry**: `docs/issues/registry.md` — all active issues with status and links.
- **Per-issue**: each folder has a `README.md` (index of what exists) and optionally an `ai-context.md` with current state and key decisions. **Read `ai-context.md` before starting work on a feature area** — it contains architectural decisions and current state that are not derivable from the code alone. Check the `Last verified:` date to gauge staleness.
- **Conventions**: `docs/issues/CONVENTIONS.md`.
